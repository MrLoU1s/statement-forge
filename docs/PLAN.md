# statement-forge — Execution Plan

> **How to use this document.** This is the sequenced, timeboxed plan for building
> statement-forge in one focused day. The executing agent (Claude Code) works **one
> milestone at a time, then STOPS** for the developer review checkpoint. Never batch
> milestones. Evidence (query counts, `EXPLAIN (ANALYZE, BUFFERS)` output, timings) is
> pasted into `PERFORMANCE.md` **verbatim, as measured** — placeholders like `[X]ms`
> remain until a real number replaces them.

---

## 0. Ground rules (apply to every milestone)

1. **Loop, not autopilot.** Finish the milestone → STOP → the developer personally runs
   the benchmark, reads the execution plan in DBeaver, and answers the "interview
   checklist" for that milestone out loud → only then commit and continue.
2. **Real numbers only.** Copy `EXPLAIN` output and timing logs straight into
   `PERFORMANCE.md` and save raw captures under `docs/evidence/`. Never summarize from
   memory. A measured "this didn't help" is a *keeper*, not a failure.
3. **Flyway owns the schema.** Never edit an applied migration (checksum error) — add a
   new `V<n>__` file. To rebuild from scratch: `docker compose down -v` (drops the
   `pgdata` volume) then restart.
4. **Keep every "before" path callable.** The statement generator takes a
   `strategy` parameter (`NAIVE`, `FETCH_JOIN`, `BATCHED`, `NATIVE_SQL`) so all
   before/after numbers stay reproducible forever. Read endpoints keep both variants
   (offset + keyset). This *is* the portfolio.
5. **Warm-cache discipline.** Run every measured query/endpoint twice; record the warm
   run (note the cold run where interesting). Numbers are laptop-local — the relative
   improvement is the claim, not the absolute value.
6. **Commit messages tell the story** (see §13). Insert real measured numbers into the
   message once captured.

## 3. M1 — Domain schema (Flyway V1) + entities (60 min)

### Build
1. `src/main/resources/db/migration/V1__baseline_schema.sql`:

   ```
   customer       (id BIGINT GENERATED BY DEFAULT AS IDENTITY PK,
                   name TEXT NOT NULL, segment TEXT NOT NULL, created_at TIMESTAMPTZ NOT NULL)
   account        (id ... IDENTITY PK, customer_id BIGINT NOT NULL FK→customer,
                   product_type TEXT NOT NULL, opened_at DATE NOT NULL, status TEXT NOT NULL)
   transaction    (id ... IDENTITY PK, account_id BIGINT NOT NULL FK→account,
                   txn_date DATE NOT NULL, amount NUMERIC(12,2) NOT NULL, description TEXT)
   statement_run  (id ... IDENTITY PK, run_date DATE, channel TEXT, status TEXT,
                   version BIGINT NOT NULL DEFAULT 0)
   statement      (id ... IDENTITY PK,          -- IDENTITY on purpose: M7 migrates it to a sequence
                   statement_run_id FK→statement_run, account_id FK→account,
                   period_start DATE, period_end DATE, total_debits NUMERIC(14,2),
                   total_credits NUMERIC(14,2), closing_balance NUMERIC(14,2), document_ref TEXT)
   comm_template  (id ... IDENTITY PK, name TEXT, channel TEXT, body_template TEXT, version INT)
   ```

   **Deliberately NO secondary indexes** — only PKs and FK constraints. Indexes arrive
   later via their own migrations, justified by measurements (that's the long-lived-app
   story). `transaction` is an unreserved keyword in PostgreSQL — legal unquoted; if
   Hibernate complains, quote it in `@Table(name = "\"transaction\"")` (it shouldn't).
2. JPA entities in `domain/`, repositories in `repo/`. All associations `LAZY`.
   `Account` has `@OneToMany(mappedBy = "account")` transactions (needed to *cause* the
   N+1 naturally). `StatementRun` gets a `@Version private long version;` field now
   (used in M9). `Statement` uses `GenerationType.IDENTITY` **on purpose** (M7 fixes it).
3. Enable measurement plumbing in `application.yaml`:
   `spring.jpa.properties.hibernate.generate_statistics: true` and log SQL at DEBUG for
   dev only (comment on how to silence it).

### Measure / verify
- `./mvnw spring-boot:run` (spring-boot-docker-compose starts Postgres automatically):
  Flyway applies V1, Hibernate `validate` passes, app boots clean.
- In DBeaver: `\d`-equivalent — six tables, correct FKs, **no secondary indexes**.

### ✋ CHECKPOINT 1 (developer)
- Boot the app yourself; open DBeaver and inspect the schema.
- Explain aloud: why `ddl-auto: validate` + Flyway (not `update`); why associations are
  LAZY; why `open-in-view: false`.
- Approve → commit.

## 5. M3 — Deliberately naive generator + baseline (70 min)

### Build
1. `StatementGenerationService` with `strategy=NAIVE`:
   - `POST /api/statement-runs?period=2025-03&strategy=NAIVE&limitAccounts=1000`
   - Creates a `statement_run` (status `RUNNING`), loads the first `limitAccounts`
     accounts (ordered by id), then **for each account**: touch
     `account.getTransactions()` (lazy → one SELECT per account, loading ALL its
     transactions), filter to the period **in Java**, sum debits/credits/closing
     balance in Java, and `statementRepository.save(...)` **one row at a time**.
   - Finish: status `COMPLETED`. Wrap in one transaction.
   - Response JSON is the benchmark record:
     `{runId, strategy, accounts, statementsWritten, elapsedMs, statementsPerSec, jdbcQueryCount}`
     (query count from Hibernate `Statistics` — inject `EntityManagerFactory`, use
     `getStatistics().getPrepareStatementCount()` cleared per run, or log it).
2. `GET /api/statement-runs/{id}` — read a run + its summary (trivial).

   **Why `limitAccounts`:** with no index on `transaction(account_id, …)`, each lazy
   load is a full scan touching a 5M-row table; a full 200k-account naive run would take
   hours. Baseline at 1,000 accounts and extrapolate linearly — and *say so* in
   `PERFORMANCE.md`. The impossibility of the naive full run at scale IS the finding.

### Measure (baseline — this feeds every later investigation)
- Run `strategy=NAIVE, limitAccounts=1000` twice; record the warm run:
  - **JDBC statement count** (expect ≈ 1 accounts query + 1,000 lazy collection loads
    + 1,000 single-row INSERTs + run bookkeeping ≈ 2,000+).
  - **elapsedMs** and **statements/sec**.
  - Linear extrapolation to 200k accounts (state as estimate).
- Keep the SQL debug log excerpt showing the same
  `select ... from transaction where account_id=?` repeated over and over.

### Evidence → PERFORMANCE.md
Create the "Baseline: the naive engine" section: symptom description, the response
JSON, the query count, the log excerpt, the extrapolation.

### ✋ CHECKPOINT 3 (developer)
- Trigger the run yourself (curl/HTTPie); watch the SQL log scroll; read the counts.
- Explain aloud: what exactly makes this 1+N (what Hibernate does at
  `getTransactions()`), and why `open-in-view: false` made it visible.
- Approve → commit.

## 6. M4 — Investigation 1: N+1 (45 min)

### Build
- `strategy=FETCH_JOIN`: repository method
  `@Query("select distinct a from Account a join fetch a.transactions t
    where a.id between :fromId and :toId and t.txnDate between :from and :to")`
  — process accounts in **id-range chunks** (e.g. 1,000 ids per chunk), *not*
  `Pageable`: fetch-join + `Pageable` triggers Hibernate's **in-memory pagination**
  (`HHH90003004`) — document this trap.
- Note the semantics: the `t.txnDate` predicate filters the *fetched collection* —
  exactly what statement generation wants, but worth one sentence in `DECISIONS.md`
  (a flat DTO projection is the stricter alternative; mention, don't build).
- Insert path unchanged (still row-by-row IDENTITY saves) — M7 owns that variable.
  One variable per investigation.

### Measure
- Same benchmark: `strategy=FETCH_JOIN, limitAccounts=1000`, warm run:
  - JDBC statement count: expect ~1,000 SELECTs collapse to ~1–2 per chunk.
  - elapsedMs before vs after (insert time still dominates — say so honestly;
    the *query-count* collapse is the headline here).

### Evidence → PERFORMANCE.md
Before/after table: strategy | JDBC queries | elapsedMs | statements/sec. Log excerpt
showing the single join query. Keep the NAIVE row visible above it.

### 🎓 Interview checklist (must be able to explain unaided)
- What triggers the lazy load; why it's 1+N and not N.
- Three fixes and when each: `JOIN FETCH` / `@EntityGraph` / DTO projection
  (+ `hibernate.default_batch_fetch_size` as the "mitigation, not fix" option).
- Why fetch-join + `Pageable` paginates in memory, and the id-range chunk alternative.
- When N+1 is actually acceptable (tiny N, second-level cache, batch-size tuned).

### ✋ CHECKPOINT 4 → approve → commit.

## 7. M5 — Investigation 2: composite index (30 min)

### Build
1. `GET /api/accounts/{accountId}/transactions?from=2025-01-01&to=2025-03-31` —
   straightforward repository query returning a DTO list (this is also the endpoint the
   on-demand statement path would use).
2. **Measure BEFORE first** (in DBeaver, twice, keep warm run):
   ```sql
   EXPLAIN (ANALYZE, BUFFERS)
   SELECT * FROM transaction
   WHERE account_id = 123456 AND txn_date BETWEEN '2025-01-01' AND '2025-03-31';
   ```
   Expect **Parallel Seq Scan**, ~5M rows filtered, large `Buffers: shared read`, and
   note `Rows Removed by Filter`. Also record API latency: `curl -w '%{time_total}'`.
3. `V3__transaction_account_date_index.sql`:
   ```sql
   CREATE INDEX idx_transaction_account_date ON transaction (account_id, txn_date);
   ```
   (Plain `CREATE INDEX` is fine here; note for the interview that production would use
   `CONCURRENTLY`, which can't run inside Flyway's transaction without extra config.)
4. Re-run the same `EXPLAIN` and the same curl → **Index Scan / Bitmap Heap Scan**,
   tiny buffer counts.

### Evidence → PERFORMANCE.md
Paste **both full plans verbatim** (also save to `docs/evidence/`), plus a row in the
summary table: `[X]ms → [Y]ms`, buffers before/after. Optional bonus: re-run the M4
benchmark once — the per-account lazy queries (still reachable via `strategy=NAIVE`)
also got faster; one sentence on investigation interaction.

### 🎓 Interview checklist
- Why `(account_id, txn_date)` in that order (equality column first, range second);
  why `(txn_date, account_id)` would be worse for this query.
- What the planner changed and *why it previously chose* a seq scan (no better path
  existed; cost model, selectivity, `ANALYZE` statistics).
- Read `Buffers:` lines: shared hit vs read; what that says about I/O.
- The cost of the index: write amplification, storage — why not index everything.
- What an index-only scan would need (covering index / `INCLUDE (amount)`).

## 8. M6 — Investigation 3: keyset vs OFFSET pagination (45 min)

### Build
1. OFFSET endpoint (naive): `GET /api/transactions?page=10000&size=50` —
   `ORDER BY id LIMIT 50 OFFSET page*50` (Spring Data `Pageable`, but use `Slice`
   or disable the count query — a `count(*)` over 5M rows on every page is its own
   mini-finding; note it).
2. Keyset endpoint: `GET /api/transactions?afterId=<cursor>&size=50` —
   `WHERE id > :afterId ORDER BY id LIMIT 50`; response includes `nextCursor`.
   (`id` is the total order and the cursor — no tie-breaker needed here; say why one
   would be needed if ordering by a non-unique column like `txn_date`.)
3. `scripts/pagination-bench.sh` (or `.ps1`): curl both endpoints at depths
   1 / 10 / 100 / 1,000 / 10,000 pages (keyset: walk the cursor, or jump straight to a
   known id ≈ depth×50), record `time_total` for each.

### Measure
- The latency-vs-depth table from the script (warm runs).
- In DBeaver:
  ```sql
  EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM transaction ORDER BY id LIMIT 50 OFFSET 500000;
  EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM transaction WHERE id > 500000 ORDER BY id LIMIT 50;
  ```
  The OFFSET plan shows it *fetched and discarded* 500k rows; the keyset plan touches ~50.

### Evidence → PERFORMANCE.md
The depth table (OFFSET column growing ~linearly, keyset flat) + both plans verbatim.

### 🎓 Interview checklist
- Why OFFSET is O(offset): the executor must produce and discard the skipped rows.
- Why keyset stays flat: index seek to the cursor, read 50.
- Requirements: stable total order (unique key or tie-breaker column), and the
  trade-off: no "jump to page 7,000" — cursor-based APIs only.
- Where this shows up in real APIs (cursor tokens in GitHub/Stripe/Slack style APIs).

### ✋ CHECKPOINT 6 → approve → commit.

## 9. M7 — Investigation 4: JDBC batch inserts (60 min)

Three measured stages — the middle one is a deliberate **negative result**.

### Stage A — before (already have it)
`strategy=FETCH_JOIN, limitAccounts=20000` warm run → insert-heavy baseline
rows/sec (row-by-row `save()`, IDENTITY ids).

### Stage B — the gotcha, measured
Set **only** `spring.jpa.properties.hibernate.jdbc.batch_size: 50` (plus
`order_inserts: true`). Re-run Stage A. **Expect: no improvement.** Hibernate
silently disables insert batching for `IDENTITY` because it must read the generated
key after each row. Capture identical rows/sec → this negative result goes in
`PERFORMANCE.md` — it is the depth the whole investigation exists to show.

### Stage C — the fix
1. `V4__statement_id_sequence.sql`:
   ```sql
   ALTER TABLE statement ALTER COLUMN id DROP IDENTITY;
   CREATE SEQUENCE statement_id_seq INCREMENT BY 50 OWNED BY statement.id;
   SELECT setval('statement_id_seq', (SELECT COALESCE(MAX(id),0)+1 FROM statement));
   ```
2. Entity: `@GeneratedValue(strategy = SEQUENCE, generator = "statement_seq")` +
   `@SequenceGenerator(name = "statement_seq", sequenceName = "statement_id_seq",
   allocationSize = 50)` (must match `INCREMENT BY`).
3. JDBC URL: append `?reWriteBatchedInserts=true` (application.yaml — keep
   compose.yaml credentials untouched).
4. `strategy=BATCHED`: same fetch-join read path; write path = accumulate,
   `saveAll(...)` + `flush()` + `clear()` every 1,000 entities (persistence-context
   hygiene at scale).
5. Verify batching is REAL: `logging.level.org.hibernate.orm.jdbc.batch: TRACE`
   (log line "Executing batch size: 50") — paste one line as proof.

### Measure
Rows/sec table with **four rows**: NAIVE · FETCH_JOIN (row-by-row) ·
batch_size-only-with-IDENTITY (**unchanged — negative result**) ·
BATCHED (sequence + batch_size + reWriteBatchedInserts). Same
`limitAccounts=20000`, warm runs.

### 🎓 Interview checklist
- *Why* IDENTITY kills batching (per-row generated-key round trip) and why SEQUENCE
  with `allocationSize` doesn't (pooled optimizer pre-allocates 50 ids in one
  `nextval`, so inserts need no immediate response).
- What `reWriteBatchedInserts` does (driver rewrites into multi-row
  `INSERT ... VALUES (...),(...)`) vs what Hibernate's `batch_size` does (JDBC
  `addBatch/executeBatch`) — two different layers.
- Id gaps are fine (sequences are non-transactional) — why that's the accepted trade.
- Why `flush()/clear()` — first-level cache growth on long transactions.

### ✋ CHECKPOINT 7 → approve → commit (message carries the ×-factor).

## 10. M8 — Investigation 5: ORM vs native SQL (45 min)

### Build
`strategy=NATIVE_SQL`: the entire monthly batch becomes **one set-based statement**
(via `JdbcTemplate` or `@Modifying` native query):

```sql
INSERT INTO statement (statement_run_id, account_id, period_start, period_end,
                       total_debits, total_credits, closing_balance, document_ref)
SELECT :runId, a.id, :periodStart, :periodEnd,
       COALESCE(SUM(t.amount) FILTER (WHERE t.amount < 0
                 AND t.txn_date BETWEEN :periodStart AND :periodEnd), 0),
       COALESCE(SUM(t.amount) FILTER (WHERE t.amount > 0
                 AND t.txn_date BETWEEN :periodStart AND :periodEnd), 0),
       COALESCE(SUM(t.amount) FILTER (WHERE t.txn_date <= :periodEnd), 0),
       'stmt-' || :runId || '-' || a.id
FROM account a
JOIN transaction t ON t.account_id = a.id
GROUP BY a.id;
```

plus a **window-function variant** for the running-balance requirement — an endpoint
`GET /api/accounts/{id}/statements/{period}/lines` returning each transaction with its
running balance:
```sql
SELECT t.id, t.txn_date, t.amount, t.description,
       SUM(t.amount) OVER (PARTITION BY t.account_id
                           ORDER BY t.txn_date, t.id) AS running_balance
FROM transaction t
WHERE t.account_id = :accountId AND t.txn_date <= :periodEnd
ORDER BY t.txn_date, t.id;
```
Try expressing the running balance in JPQL first — you can't (no window functions) —
and record that dead end in `DECISIONS.md`. That's the justification, demonstrated.

### Measure
- `strategy=NATIVE_SQL` for the **full 200k accounts**, one month — elapsed +
  statements/sec (this full run was *impossible* naive, painful batched; likely seconds
  native — the arc completes).
- Same scale comparison: BATCHED vs NATIVE_SQL at `limitAccounts=20000`, warm.
- `EXPLAIN (ANALYZE, BUFFERS)` of the SELECT body in DBeaver (note the index from M5
  appearing — or not, if the planner prefers a seq scan for a full-table aggregate;
  either outcome is good material — explain *why* the planner is right).

### Evidence → PERFORMANCE.md
Timing table (BATCHED vs NATIVE at 20k; NATIVE at 200k), the plan, one paragraph:
row-by-row entity materialization + network round trips vs set-based work where the
data lives. `DECISIONS.md`: when I'd still choose JPA (OLTP CRUD, small object graphs,
type-safety) vs native (bulk set operations, window functions, reporting).

### 🎓 Interview checklist
- What an entity costs (materialization, dirty checking, 1st-level cache) × 200k.
- Window function semantics: `PARTITION BY` / `ORDER BY` / frame — explain the
  running-balance query line by line.
- Why this can't be JPQL; options that keep native SQL maintainable (named queries,
  constructor/interface projections, Flyway-owned schema, integration tests).
- `INSERT ... SELECT` transactional behavior — all-or-nothing in one statement.

### ✋ CHECKPOINT 8 → approve → commit.

## 11. M9 — Investigation 6: optimistic locking (30 min)

### Build
1. `@Version` already on `StatementRun` (M1). Status transition endpoint:
   `PATCH /api/statement-runs/{id}` body `{"status": "APPROVED"}` — read, mutate, save.
2. Map `ObjectOptimisticLockingFailureException` → HTTP `409 Conflict`
   (`@RestControllerAdvice`).
3. Demonstration (pick ONE, timebox hard):
   - **Integration test** (Testcontainers): two `TransactionTemplate` blocks load the
     same run (same version), both modify; second commit throws — assert it. OR
   - Two concurrent curl PATCHes primed by a short artificial delay.
4. Write the **isolation note** in `DECISIONS.md` (~10 lines): the batch reads
   immutable historical transactions and inserts new rows, so PostgreSQL's default
   `READ COMMITTED` suffices; the header's lost-update risk on status is covered by the
   version column; when `REPEATABLE READ` would matter (multi-query consistency over
   concurrently-changing data) and what serialization failures would force (retry).

### Measure / evidence
The test output / the 409 response + the `UPDATE ... SET version=? WHERE id=? AND
version=?` SQL line from the log, pasted into `PERFORMANCE.md` (short section).

### 🎓 Interview checklist
- Mechanics: version in the `WHERE` clause; 0 rows updated → exception.
- Optimistic vs pessimistic (`SELECT ... FOR UPDATE`): contention profile, deadlock
  risk, user experience (retry vs block).
- The lost-update anomaly this prevents; PG isolation levels and the default.

## 15. Known gotchas (pre-empted roadblocks)

1. **Flyway checksum**: never edit an applied migration; new `V<n>__` file, or
   `docker compose down -v` to rebuild the DB in dev.
2. **IDENTITY silently disables Hibernate insert batching** — the M7 centerpiece.
3. **Fetch join + `Pageable` = in-memory pagination** (`HHH90003004`) — chunk by id
   range instead.
4. **`Page` fires `count(*)` over 5M rows per request** — use `Slice`/no-count; worth a
   sentence in PERFORMANCE.md.
5. **`transaction` is an unreserved keyword** in PG — fine unquoted; quote in `@Table`
   if anything complains.
6. **`CREATE INDEX CONCURRENTLY` can't run in a transaction** — plain `CREATE INDEX`
   in Flyway here; mention the production difference in DECISIONS.md.
7. **First boot seeds for 1–3 min** — don't kill it; watch Flyway logs.
8. **Warm vs cold cache** — run twice, record warm, note cold where interesting.
9. **Windows**: run `./mvnw` in Git Bash or use `mvnw.cmd`; CRLF warnings from git are
   harmless.
10. **spring-boot-docker-compose lifecycle**: local app start auto-starts compose
    services; the `app` container (M10) needs the ignore label to avoid recursion.
11. **Persistence-context bloat** on big writes — `flush()/clear()` every chunk.
12. **JDBC URL change (`reWriteBatchedInserts`) belongs in `application.yaml`** — do
    not touch compose credentials (must stay in sync).
